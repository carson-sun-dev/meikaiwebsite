## 2. 关键技术决策

| 决策 | 选择 | 理由 |
|---|---|---|
| RAG 数字暴露策略 | `format_examples_for_prompt` **显式不打 subtotal/unit_price** | Guardrail 输出侧只允许 quote_block 内的金额;若 prompt 暴露案例金额,L2 复述会被判违规。素材只给类目+spec,金额由 quote_block 兜底 |
| RAG business_line 过滤位置 | 客户端 post-filter(项目维度映射) | quote_items_bge payload 无 business_line(line item 维度);改 schema 需全量 re-index 成本高;projects_labeled.jsonl 加载 `project_id→business_line` 映射后客户端过滤,粗召回 top_k*4 留余量 |
| RAG 异常降级 | 双层 try(rag.py 内部 + subgraphs 外层) | rag.py 内 catch-all 是显式契约;subgraphs 外层 try 是防御性兜底(以后有人误删内部 try 时,SubGraph 流不崩) |
| RAG import 路径 | `TYPE_CHECKING` 引入 `BusinessLine` | `app.graphs.__init__` 顶层 import subgraphs,subgraphs import rag → 运行期 import `app.graphs.state` 会循环;Literal 类型别名只用于注释,TYPE_CHECKING 块完全解决 |
| Checkpointer 工厂降级 | MySQL 不可达 → 静默回落 InMemorySaver + WARN | 简历级项目:MySQL 持久化体现"生产就绪",mock/dev 没起 mysql 时仍能跑;严格要求 MySQL 会抬高新人/CI 起步门槛 |
| Multi-turn 槽位 monotonic 合并 | `_merge_slots`:已知字段保留、新 truthy 覆盖、`is_quote_intent=True` 锁定 | LLM 第二轮只看本轮短消息("200 平米")会抽不到业态,朴素 dict.update 会抹掉上一轮 → 违反用户预期;monotonic 让对话状态只增不减,DESIGN §11.4 工程纪律 |
| Router 短路条件 | `state.business_line` 非空且非 `unknown` 才短路 | unknown 是兜底失败信号(L0 误判),保留再调 L0 的机会;有效分类则跳过省 token + 防短消息("200 平米")被 router 误判 |
| 集成测试用账号 | meikai 账号 + 现有 meikai 库 | mysql 8.0 默认 root@localhost(socket only),从 host TCP 进会被拒;改 docker-compose 加 MYSQL_ROOT_HOST=% 暴露生产 root 远程入口风险大;meikai@% 已是 docker entrypoint 默认创建,够用 |
| 集成测试库选择 | 不建独立库,共享 meikai 库,前后清场 4 张表 | meikai 用户没 CREATE DATABASE 权限;langgraph 自管理表(checkpoints / checkpoint_blobs / checkpoint_writes / checkpoint_migrations)与 backend 表命名空间不冲突;前后清场不动开发数据 |
| MySQL 不可达时行为 | autouse fixture ping → fixture 内二次 ping → 优雅 skip + 重建命令提示 | error 红栈让开发者以为代码挂了;skip + 修复指令更人性 |
| pytest marker 策略 | `addopts="-m 'not integration'"` 默认排除 | 单测应秒过零依赖;集成显式触发(`pytest -m integration`),CI 短期内只跑单测 |

## 7. 给未来自己的踩坑提醒

1. **mysql_data 卷密码与 .env 漂移**:本机这次 mysql 容器是 2026-04 创建的,卷内密码与现 `.env` 不同步。集成测试 fixture 已优雅 skip,但要真跑必须先 `docker volume rm meikai_website_mysql_data`。生产部署初次启动时要确保 `.env` 与卷一次性对齐,否则后期改 `.env` 不生效。
2. **aiomysql 连 mysql 8.0 需要 cryptography**:caching_sha2_password 是默认认证;`pyproject.toml` 现在没把 `cryptography` 列依赖,本机是手动 `uv pip install` 装的。如果 CI 跑集成测试,需要把它写进 `[project.dependencies]` 或一个 `[project.optional-dependencies].integration` extras。
3. **backend 那张 ai_checkpoints 表其实没用**:`backend/src/db/migrate.ts:40` 建的 `ai_checkpoints(session_id/thread_ts/state_json)` 是历史预留;langgraph-checkpoint-mysql 3.0.0 的 `setup()` 自建 4 张表(`checkpoints` / `checkpoint_blobs` / `checkpoint_writes` / `checkpoint_migrations`)。**不要去删 backend 这张表**(没造成困扰,留给文档说明);也**不要尝试用它做手工 checkpoint**(schema 不匹配)。集成测试 docstring + README 都已显式说明。
4. **RAG 第二层 try 不可删**:`subgraphs.py` 在调 `retrieve_quote_examples` 外层包了一个 try。`rag.py` 内部已 catch-all 降级,但外层 try 是防御性兜底 — 未来如果有人重构 rag.py 内部去掉 catch,subgraph 报价主路径仍然不该崩。两层 try 各有契约,均保留。
5. **router 短路对 mock 模式的影响**:mock 路径下 L0 router 总把消息分类成第一个出现的标签(residential),所以第二轮"200 平米"短路后业务线仍是 residential,与第一轮一致(虽然第一轮就是 residential 误判)。这是 mock 固有局限,不要为此改 router 逻辑;live 模式无此问题。
6. **`is_quote_intent` 锁定的潜在边界**:`_merge_slots` 让 `is_quote_intent=True` 一旦设置就永远 True。如果用户对话演进到"问完价之后聊别的",依然会落报价分支。这对营销引流场景合理(用户问过价 = 高意向,导购话术应延续);但若未来要做"严格上下文切换",需要加 `intent_decay` 或 turn counter。当前不做。
