二、 面试极大概率被深挖的“提升点”（防御准备）
大厂面试官在看到这么完善的设计后，一定会进行压力测试（Stress Test），主要会集中在以下几个方向。建议你在项目实施或复盘阶段提前准备好答案：

1. 高并发与异步处理机制（2C2G 的阿喀琉斯之踵）

痛点： 2C2G 资源极度有限。当多个用户同时上传 CAD 文件（ezdxf 解析是 CPU 密集型任务）或同时发起长轮询检索时，FastAPI 很容易阻塞甚至 OOM（内存溢出）。

提升方案 / 面试答法： * 解释在 FastAPI 中如何严格区分 CPU 密集型任务（CAD 解析、PaddleOCR）和 I/O 密集型任务（调用 LLM API）。

可以引入轻量级的任务队列（如果 Redis 跑不动，可以考虑基于 Python 内置的 asyncio.Queue 做简单的内存队列限流，或者限制并发 worker 数量）。

明确系统熔断策略：当系统负载过高时，优雅降级（直接跳过图纸解析，引导用户走纯文本流）。

2. 数据飞轮与长效评估闭环（Data Flywheel）

痛点： 文档提到了 30 条评测集（LLM-as-judge），但这属于静态评测。线上系统上线后，如果模型报了一个离谱的价格，系统怎么学习纠正？

提升方案 / 面试答法： * 设计反馈机制：前端加入点赞/踩，或者内部销售人员发现报价不准时，一键标记“Bad Case”。

形成闭环：被标记的 Bad Case 通过自动化脚本提取，经过人工修正后，自动写入到 Qdrant 的 Few-shot 池（L4 Context）中。告诉面试官：“我的系统具备自我进化能力，越用越准”。

3. State 的持久化与会话恢复

痛点： LangGraph 的 StateGraph 默认是在内存中流转的。如果用户聊到一半关闭了网页，或者你的 FastAPI 容器重启了，用户的槽位（Slot）数据还在吗？

提升方案 / 面试答法： * 既然你已经有了 MySQL 存放现有网站数据，可以用它（或 SQLite 轻量级）作为 LangGraph 的 Checkpointer 存储后备。实现会话状态的持久化（Persistence），支持用户明天打开网页接着填槽位。

4. 检索系统的量化基准（Quantitative Metrics）

痛点： 文档中提到“召回 NDCG@5 提升 ~25%”，面试官一定会问：“这个 25% 是怎么算出来的？你的 Baseline 是什么？”

提升方案 / 面试答法： * 准备好具体的数据支撑。例如：“我的 Baseline 是纯 BM25 检索，评测集是根据真实客户提问构造的 50 个 Query。纯 BM25 在专有名词上经常 miss，加上稠密向量和 RRF 重排后，前 5 个结果中包含正确报价案例的比例从 60% 提升到了 85%。”
