## 2. 关键技术决策

| 决策 | 选择 | 理由 |
|---|---|---|
| office INDUSTRY_DEFAULT 数值 | 历史 P10=778 / P90=1000(×0.9 后显示价严格在 ¥700-900) | 店主要求"控制在 700-900 每平米",显示价(经 `_round_yuan` ¥100 取整)三档差小但合规;premium 段塌缩到 ¥900 单点,可接受 |
| residential INDUSTRY_DEFAULT 数值 | 历史 P10=500 / P90=1400(下限不变,上限大幅收窄) | 店主认为原 ¥500-2500 跨度过宽;新口径显示价 basic ¥400-600 / mid ¥600-1000 / premium ¥1000-1300,三档梯度清晰 |
| 槽位提取双层(L0 LLM + 正则) | 既调 L0 抽 JSON 又跑正则兜底,LLM 字段优先合并 | L0 在中文长尾("两百来平"/"90 个平方")更鲁棒,但偶发解析失败;正则保证 mock 模式 + L0 异常都不让报价分支坍塌(DESIGN §11.4 工程纪律) |
| 业态正则匹配顺序 | "长在前":火锅店/茶馆 必须先于 饭店/店 | 否则 "我开个火锅店" 会抽成 business_type="饭店" 或 "店" |
| 报价 prompt 注入策略 | quote_block 标签包装 + 严格规则("原样引用 + 前后可加 1-2 句导购") | 单次 L2 调用维持流式体验;Guardrail 兜底数字合规,prompt 约束是 best-effort |
| SubGraph 引导分支 prompt | 注入已抽到的部分槽位 + 缺失槽位清单 | 避免 L2 重复问已知的(用户体验);列出缺失让 L2 知道该追问什么 |
| Guardrail 触发后行为 | **追加** "[系统校准]" 段而非覆盖整文 | 覆盖会让用户已看到的前段消失造成困惑;追加保留过程感同时提供 fallback,合规问题不暴露给用户 |
| Guardrail 金额抽取 regex | 必须有 ¥ 或 元 锚定才计入金额(`(?:¥\s*([\d,]+)\|([\d,]+)\s*元)`) | 否则 "200㎡"/"52.8%"/"3 室" 会被误判为金额;LLM 用"约 1 万"表达会漏检 1<100 阈值,但"没编新数字"本身合规,接受漏检 |
| Guardrail 比较口径 | 千分位 `¥56,000` 与裸 `56000` 等价 | quote 工具产出无逗号,LLM 输出有,不能错杀;统一去逗号后整数比较 |

## 7. 给未来自己的踩坑提醒

1. **mock 路由分类被回显带偏**:`MockLLM` 的回复模板把 prompt 前 80 字回显,`ROUTER_PROMPT` 第二段恰含 "residential / office / storefront" 字面,导致 mock 路径下 router 总命中第一个出现的标签(residential)。Live 模式无此问题,但 mock 端到端测试时要意识到 business_line 可能不准 — 这是 mock 的固有局限,**不要为此改 router 逻辑**。
2. **`_round_yuan` 取整粒度**:¥100 取整 + Python banker's rounding 让 700-900 这种窄区间在显示侧三档塌缩。如果客户后续给"更窄"的口径(比如 ¥800-850),需要先用 `python -c "from app.tools.quote import estimate_office_quote; ..."` 反推实际显示效果再答应,不要按字面值直接写 INDUSTRY_DEFAULT。
3. **Guardrail 追加段的 newline**:`GUARDRAIL_CORRECTION_PREFIX` 以 `\n\n———\n` 开头,前端如果做 markdown 渲染要确保 `———` 不被转成横线分隔符破坏布局(SSE 是逐字流出,这段也会逐字推)。
